# HuggingChat iOS

Hybrid iOS app that wraps the SvelteKit chat UI in a `WKWebView` and provides
native chrome (tab bar, conversation list, model picker, settings, share, file
picker) on top.

The streaming message renderer (markdown, code highlighting, tool calls,
reasoning, citations, math) lives entirely in the WebView so the iOS app
inherits parity with the web experience for free. Native code only owns the
shell.

## Architecture

```
SwiftUI shell (RootTabView)
  ├─ Chat tab
  │   └─ ChatWebView (WKWebView)
  │       └─ loads /(embed)/conversation/[id] from huggingface.co/chat
  │           ↳ JS bridge ⇄ Swift via window.webkit.messageHandlers.hfchat
  ├─ History tab → GET /api/v2/conversations
  └─ Settings tab → GET /api/v2/user/settings
```

See `/CLAUDE.md` and the design doc for the full plan.

## Building

The Xcode project is generated by [XcodeGen][xg], not committed. To open the
project:

```sh
brew install xcodegen
cd ios
xcodegen generate
open HuggingChat.xcodeproj
```

Then build and run on an iPhone simulator (Xcode 16+, iOS 17+ deployment
target).

[xg]: https://github.com/yonaskolb/XcodeGen

## Testing against a local SvelteKit dev server

The app hardcodes `https://huggingface.co/chat` as the origin. For local
development against `npm run dev`:

1. The session cookie is `SameSite=None; Secure`, so it requires HTTPS even in
   dev. Either:
   - Test against production (`huggingface.co/chat`) for full integration runs.
   - Or run Vite over HTTPS with a self-signed cert and add it to the
     simulator's trusted roots.
2. Update `AppConfig.baseURL` to point at your local server.
3. Confirm the embed route renders at `/(embed)/conversation/<id>` in a desktop
   browser before loading it in the simulator.

## Auth flow

1. App launch reads `WKWebsiteDataStore.default()` for an unexpired `hf-chat`
   session cookie at `huggingface.co`.
2. If absent: tap **Sign in** → `LoginCoordinator` presents
   `ASWebAuthenticationSession` at `…/chat/login?next=/chat/login/native-done`.
3. The OAuth provider redirects through `…/chat/login/callback` (which writes
   the session cookie to `HTTPCookieStorage.shared` because the auth session
   isn't ephemeral) and lands on `…/chat/login/native-done`.
4. That route returns an HTML page with `<meta http-equiv="refresh"
content="0;url=hf-chat://auth-done">`. The custom-scheme jump fires the
   `ASWebAuthenticationSession` completion handler.
5. `CookieJar.syncSharedToWebView()` copies the session cookie into the
   WebView's cookie store. `AuthState.isLoggedIn` flips and the chat tab
   appears.

## JS bridge contract

The web side is in `src/lib/native/bridge.ts`. The Swift side decodes the same
union type in `Bridge/BridgeMessage.swift`. Keep both in sync — adding a new
message means editing both files plus `BridgeDispatcher`.

## Out of scope for v1

Push notifications, native voice input, MCP server management UI, share
extension, widgets / Live Activities, offline message queueing, self-hosted
instance picker, universal links.
